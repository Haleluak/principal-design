# Design S3 Upload & CloudFront Delivery System

Tài liệu này hướng dẫn cách thiết kế hệ thống upload ảnh lên AWS S3 và phân phối qua CloudFront một cách tối ưu, bảo mật và hiệu năng cao.

## 1. Kiến trúc tổng quan

Dưới đây là mô hình hoạt động tiêu chuẩn (Best Practice):

### Sơ đồ luồng hoạt động (Workflow)

```text
GIAI ĐOẠN UPLOAD:
Client (Web/App) ----(1) Yêu cầu upload----> Backend API
Backend API      ----(2) Xác thực & Xin URL--> AWS S3
Backend API      <---(3) Trả về Presigned URL- AWS S3
Client (Web/App) <---(4) Nhận Presigned URL-- Backend API
Client (Web/App) ----(5) Upload file trực tiếp lên S3 (Dùng URL bước 4)

GIAI ĐOẠN TRUY XUẤT (VIEWING):
User/Client ----(6) Truy cập cdn.domain.com/anh.jpg ----> CloudFront (CDN)
CloudFront  ----(7) Kiểm tra Cache (Nếu có trả về luôn) -> Cache Layer
CloudFront  ----(8) Nếu không có, lấy từ S3 (OAC) ------> AWS S3
User/Client <---(9) Nhận ảnh tốc độ cao từ Edge -------- CloudFront
```

**Chi tiết các bước:**
1.  **Upload:** Client không gửi file qua Backend (để tránh làm chậm server). Backend chỉ đóng vai trò "người gác cổng" để kiểm tra quyền và xin S3 một cái "vé" (Presigned URL) cho phép Client upload trong một khoảng thời gian ngắn.
2.  **Lưu trữ:** File nằm trong S3 Bucket được khóa kín (Private). Không ai có thể xem trực tiếp từ link S3.
3.  **Phân phối:** Khi cần xem ảnh, Client gọi qua CloudFront. CloudFront có quyền "đặc biệt" (OAC) để vào S3 lấy ảnh, sau đó lưu lại bản sao tại các máy chủ gần người dùng nhất để lần sau tải cực nhanh.

## 2. Tại sao phải sử dụng CloudFront?

Việc để User truy cập trực tiếp vào S3 là một "bad practice" trong môi trường Production. CloudFront giải quyết các vấn đề sau:

1.  **Hiệu năng (Latency):** CloudFront có hơn 400+ Edge Locations trên toàn cầu. Khi user ở Việt Nam truy cập, họ sẽ lấy ảnh từ Edge gần nhất thay vì phải kéo từ Region (ví dụ `us-east-1` bên Mỹ).
2.  **Bảo mật với OAC (Origin Access Control):** Bạn có thể đóng hoàn toàn quyền truy cập Public của S3. Chỉ CloudFront mới có quyền đọc file từ S3. Điều này ngăn chặn việc leak "S3 URL" và bảo vệ data.
3.  **Hỗ trợ HTTPS & Custom Domain:** S3 bản thân không hỗ trợ HTTPS cho tên miền riêng (`cdn.yourcompany.com`). CloudFront cung cấp chứng chỉ SSL miễn phí qua AWS Certificate Manager (ACM).
4.  **Tối ưu chi phí:** Phí Data Transfer Out từ CloudFront thường rẻ hơn từ S3 trực tiếp (đặc biệt là với gói Free Tier hoặc lưu lượng lớn).
5.  **Image Transformation (Nâng cao):** Có thể kết hợp CloudFront Functions hoặc Lambda@Edge để resize ảnh, nén ảnh (WebP) real-time.

## 3. Quy trình Setup chi tiết

### Bước 1: Cấu hình S3 Bucket
1.  **Create Bucket:** Đặt tên unique (ví dụ: `my-app-assets`).
2.  **Block Public Access:** Bật (ON) tất cả các options để bucket hoàn toàn không thể truy cập từ Internet.
3.  **CORS Configuration:** Cực kỳ quan trọng để Client có thể upload trực tiếp.
    ```json
    [
        {
            "AllowedHeaders": ["*"],
            "AllowedMethods": ["PUT", "POST", "GET"],
            "AllowedOrigins": ["https://your-domain.com"],
            "ExposeHeaders": ["ETag"]
        }
    ]
    ```

### Bước 2: Cấu hình CloudFront OAC
1.  **Create Distribution:** Chọn Origin là S3 Bucket vừa tạo.
2.  **Origin Access:** Chọn "Origin access control settings (recommended)".
3.  **Create Control Setting:** Chọn "S3" và để mặc định (Sign requests).
4.  **Bucket Policy:** Sau khi tạo xong, CloudFront sẽ cung cấp một đoạn JSON Bucket Policy. Bạn phải **Copy** đoạn này và dán vào phần **S3 Bucket -> Permissions -> Bucket Policy**.
    *Nó sẽ có dạng: Cho phép `Service: cloudfront.amazonaws.com` thực hiện `s3:GetObject`.*

### Bước 3: Backend Logic (Presigned URL)
Thay vì upload file qua Server (làm nghẽn băng thông server), hãy dùng **Presigned URL**.

**Ví dụ Python Best Practices (Sử dụng Boto3):**

```python
import logging
import boto3
from botocore.exceptions import ClientError
from typing import Dict, Optional, Any
import uuid
import os

# Cấu hình logging chuyên nghiệp
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

def generate_secure_upload_url(
    bucket_name: str,
    file_extension: str,
    content_type: str,
    user_id: str,
    expiration: int = 300  # Ngắn (5 phút) để tăng bảo mật
) -> Optional[Dict[str, Any]]:
    """
    Tạo Presigned URL với các ràng buộc bảo mật chặt chẽ.
    Sử dụng UUID để tránh va chạm tên file và Path Traversal.
    """
    s3_client = boto3.client(
        's3',
        region_name=os.getenv('AWS_REGION', 'ap-southeast-1'),
        config=boto3.session.Config(signature_version='s3v4') # Bắt buộc dùng SigV4
    )

    # 1. Tạo tên file an toàn (Security: Tránh User Control Filename)
    safe_filename = f"uploads/{user_id}/{uuid.uuid4()}.{file_extension.strip('.')}"

    try:
        # Sử dụng generate_presigned_url cho phương thức PUT
        # Nếu muốn giới hạn File Size ngay tại S3, hãy cân nhắc dùng generate_presigned_post
        params = {
            'Bucket': bucket_name,
            'Key': safe_filename,
            'ContentType': content_type,
            # 'ACL': 'private' # Mặc định là private nếu bucket đã block public access
        }

        url = s3_client.generate_presigned_url(
            ClientMethod='put_object',
            Params=params,
            ExpiresIn=expiration
        )

        logger.info(f"Generated secure URL for {safe_filename}")
        
        return {
            "upload_url": url,
            "file_path": safe_filename,
            "expires_in": expiration
        }

    except ClientError as e:
        logger.error(f"Error generating presigned URL: {e}")
        return None

# Cách sử dụng nâng cao:
# - Kiểm tra Content-Type hợp lệ (Whitelist) trước khi gọi hàm này
# - Kiểm tra Quyền (RBAC) của user_id trước khi cấp URL
```

## 4. Best Practices "Kỹ nhất"

1.  **Cấu trúc thư mục (Prefix):** Không nên để tất cả ảnh trong root. Nên chia theo `user-id/YYYY/MM/DD/filename.jpg` để tối ưu indexing của S3.
2.  **Versioning:** Bật S3 Versioning để tránh việc ghi đè nhầm hoặc xóa nhầm data.
3.  **Lifecycle Rules:** Thiết lập rule để tự động chuyển các ảnh cũ (ví dụ 1 năm không ai xem) sang **S3 Intelligent-Tiering** hoặc **Glacier** để tiết kiệm tiền.
4.  **Cache Policy (CloudFront):** Sử dụng `CachingOptimized` và cấu hình Header `Cache-Control: public, max-age=31536000` khi upload để trình duyệt cache ảnh lâu nhất có thể.
5.  **Security:** Luôn validate `FileType` và `FileSize` ở Backend trước khi cấp Presigned URL để tránh bị user upload virus hoặc file quá lớn phá hoại bucket.

## 5. Các Edge Cases & Lưu ý cho Principal Engineer

Để đạt đến trình độ hệ thống chịu tải cao và tin cậy tuyệt đối, bạn cần chú trọng các điểm sau:

### A. Xử lý File lớn (Large File Uploads)
*   **Vấn đề:** Presigned URL thông thường có giới hạn và dễ lỗi nếu upload file hàng GB.
*   **Giải pháp:** Sử dụng **S3 Multipart Upload**. Backend sẽ khởi tạo upload, trả về một `UploadId` và một danh sách Presigned URLs cho từng "part". Client upload song song các part và yêu cầu Backend "Complete" sau khi xong.

### B. CDN Invalidation vs Fingerprinting
*   **Vấn đề:** Khi bạn sửa 1 cái ảnh nhưng giữ nguyên tên file `avatar.jpg`, CloudFront sẽ vẫn trả về ảnh cũ do còn cache.
*   **Giải pháp:** 
    *   **Tốt nhất:** Dùng "Filename Versioning" hoặc "Content Hashing" (ví dụ: `avatar.v2.jpg` hoặc `avatar.a1b2c3.jpg`).
    *   **Tránh:** Dùng `Invalidation` thường xuyên vì nó tốn phí (Free 1000 path/tháng) và có độ trễ.

### C. Giới hạn của S3 (Prefix Throttling)
*   **Vấn đề:** S3 có giới hạn 3,500 PUT/5,500 GET request mỗi giây trên mỗi **Prefix** (thư mục). Nếu hàng triệu user upload vào cùng 1 folder, S3 sẽ báo lỗi 503 Slow Down.
*   **Giải pháp:** Tạo cấu trúc thư mục ngẫu nhiên ở đầu (ví dụ: `/a1/image.jpg`, `/b2/image.jpg`). Điều này giúp S3 phân phối tải trọng ra nhiều phân vùng vật lý khác nhau.

### D. Security: Presigned URL Leakage
*   **Vấn đề:** Nếu URL bị lộ, ai cũng có thể upload file rác vào bucket của bạn.
*   **Giải pháp:**
    *   Set `ExpiresIn` cực ngắn (ví dụ 60-300 giây).
    *   Validate `Content-Length` và `Content-Type` ngay khi generate URL để S3 từ chối nếu file thực tế không khớp.

### E. Cost Optimization (Tối ưu chi phí)
*   **Vấn đề:** Chi phí lưu trữ có thể phình to nếu không kiểm soát ảnh rác hoặc ảnh cũ.
*   **Giải pháp:**
    *   **S3 Intelligent-Tiering:** Tự động chuyển file ít dùng sang vùng rẻ tiền.
    *   **Lifecycle Rules:** Tự động xóa các file upload dở dang (Multipart Uploads) sau 7 ngày để tránh tốn tiền dung lượng cho file lỗi.

### F. Observability (Giám sát)
*   **Vấn đề:** Không biết tại sao user báo upload lỗi.
*   **Giải pháp:** 
    *   Bật **S3 Server Access Logging** và **CloudFront Logs**.
    *   Theo dõi các metrics 4xx/5xx trên CloudWatch để nhận diện sự cố ngay lập tức.
